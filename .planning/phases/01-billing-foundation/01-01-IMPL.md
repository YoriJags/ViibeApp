# Phase 1 — Billing Implementation Inventory
*Recorded: 2026-03-28*

## Overview

The VIIBE billing system is built entirely on Paystack for both consumer subscriptions (Vibe+, ₦2,000/month) and merchant wallet top-ups. No Flutterwave-specific files exist anywhere in the backend; a grep search of all `.py` files in `backend/app/` returned no matches for "flutterwave". RevenueCat (iOS/Android IAP) is not integrated — the SDK is absent from `frontend/package.json` and no RevenueCat imports exist in the codebase. Merchant SaaS subscription (₦150,000/month) has no dedicated backend route file; it is not implemented.

## Backend Files

### backend/app/services/payments.py
**Role:** Paystack payment helpers — webhook HMAC-SHA512 signature verification, bank account resolution, transfer recipient creation, and transfer initiation. Focused on scout coin cashout flow, not subscription flow.
**Key functions:**
- `verify_paystack_signature(payload: str, signature: str) -> bool` — HMAC-SHA512 verification using `PAYSTACK_SECRET_KEY`. Returns `True` (skip check) if key is not configured.
- `resolve_bank_account(account_number, bank_code) -> dict` — Paystack `/bank/resolve` call; timeout=10s.
- `create_transfer_recipient(name, account_number, bank_code) -> str` — Creates Paystack transfer recipient; timeout=10s.
- `initiate_transfer(recipient_code, amount_kobo, reason) -> dict` — Initiates Paystack transfer; timeout=15s.
- `get_banks() -> list` — Lists Nigerian banks from Paystack; timeout=10s.
**Constants:** None billing-specific. References `PAYSTACK_SECRET_KEY` from `app.config`.
**Notes:** No try/except wrapping the httpx calls in `resolve_bank_account` or `create_transfer_recipient`. `initiate_transfer` also lacks try/except; exceptions propagate to caller. Only `get_banks` silently returns `[]` on failure. The timeout parameters are present on these helper functions but no network error handling exists.

### backend/app/routes/subscriptions.py
**Role:** Vibe+ subscription flow — initialize Paystack payment, verify payment, activate subscription, return status. Also provides `_check_vibe_plus()` helper imported by other routes.
**Key functions/routes:**
- `POST /api/subscription/initialize` — Creates a `pending_subscriptions` DB record, calls Paystack `/transaction/initialize`; returns `authorization_url` and `reference`. Auth required.
- `POST /api/subscription/verify/{reference}` — Calls Paystack `/transaction/verify/{reference}`, checks amount >= expected, calls `_apply_subscription()`. Atomic claim via status field. Auth required.
- `GET /api/subscription/status` — Returns `is_vibe_plus`, `expires_at`, `days_remaining` for authenticated user.
- `_check_vibe_plus(user: dict) -> bool` — Checks `is_vibe_plus` flag and `vibe_plus_expires_at`; silently expires in DB if past.
- `_apply_subscription(user_id, reference) -> dict` — Sets `is_vibe_plus=True`, `vibe_plus_expires_at = now+30days`, writes to `platform_revenue` collection.
- `verify_subscription_by_reference(reference) -> None` — Webhook-callable; looks up `pending_subscriptions` by reference, atomic claim, calls Paystack verify, activates subscription.

**Price constant:** `DEFAULT_PRICE_KOBO = 200000` (= ₦2,000). Price is also read dynamically from `platform_settings` collection via `get_platform_setting("vibe_plus_price_kobo")`; `DEFAULT_PRICE_KOBO` is the fallback.

**Trial support:** No — `_apply_subscription` sets `expires_at = now + 30 days` unconditionally. No `start_date` or trial delay sent to Paystack. No `trial_ends_at` field stored.

**Network handling on initialize:** The `POST /transaction/initialize` call has `timeout=15` and is wrapped in `try/except` catching `HTTPException`, `httpx.TimeoutException`, and general `Exception`. Returns HTTP 504 on timeout, HTTP 502 on non-200 from Paystack, HTTP 500 on general error. **This call is already hardened.**

**Network handling on verify (user-facing):** The `GET /transaction/verify/{reference}` call in `verify_subscription()` has `timeout=10` but the `except Exception` block resets pending status and raises HTTP 400 with `str(e)` — this exposes raw exception messages to the client on network failure.

**Network handling on verify (webhook-callable):** `verify_subscription_by_reference()` has `timeout=10` but general `except Exception` only logs and marks status="failed". Does not crash.

**Notes:** Reference prefix is `VIIBE-PLUS-` (double-I). Webhook handler checks for `VIIBE-PLUS-` prefix. The `_apply_subscription` function reads `price_kobo` from `platform_settings` for the revenue log — it does NOT validate against the amount received from Paystack at this step (validation happens in the verify endpoint, not in `_apply_subscription`).

### backend/app/routes/webhooks.py
**Role:** Single Paystack webhook handler. Routes events by reference prefix to wallet top-up or Vibe+ subscription flows.
**Registered events handled:**
- `charge.success` — Inspects `data.reference`:
  - Prefix `VIBE-TOPUP-` → calls `verify_wallet_topup(reference)` (from merchant routes)
  - Prefix `VIIBE-PLUS-` → calls `verify_subscription_by_reference(reference)` (from subscriptions)
  - All other references are silently ignored (no log, no error)

**Signature verification:** Calls `verify_paystack_signature(body.decode(), signature)` from `payments.py`. On failure: returns HTTP 401 with `{"status": "unauthorized"}`. On success: proceeds to event routing.

**Logging on failure:** `logger.warning("Invalid Paystack webhook signature")` is called on failure. However, it logs NO context — no IP address, no received signature prefix, no event type. This provides minimal diagnostic value.

**Notes:** No handling for `subscription.create`, `invoice.payment_failed`, or `subscription.disable` events. No Flutterwave webhook handler. Returns HTTP 200 `{"status": "ok"}` for all valid signatures regardless of whether the reference was recognized.

### backend/app/routes/planner.py
**Route:** `POST /api/planner/chat`
**Role:** Night Planner AI concierge — Claude AI with rule-based fallback. Accepts city, message, history, conversation_id, user_id.
**Vibe+ gate:** **Absent.** No call to `_check_vibe_plus()`, no auth check. The route has no `Request` parameter and does not require a Bearer token. Any unauthenticated caller can use the Night Planner without a Vibe+ subscription.
**Notes:** The frontend `NightPlannerModal.tsx` (actually named `NightPlannerModal.tsx` but contains `AskVibeModal` component) calls `/api/planner/chat` and does not check `isVibePlus()` before opening — no gate at the UI level either.

### backend/app/routes/pulse_drops.py
**Route:** `POST /api/pulse-drops/purchase`
**Role:** Purchase Pulse Drop boost using merchant wallet balance. Handles tier selection, wallet deduction, revenue credit, pulse drop creation, venue update, and real-time broadcast.
**SaaS subscription gate:** **Absent.** The route only checks wallet balance (returns HTTP 402 "Insufficient balance"). There is no check for an active merchant SaaS subscription before allowing a Pulse Drop purchase. No auth/identity check either — `drop_data.venue_id` is passed in the request body with no ownership verification.
**Notes:** Also provides `GET /api/pulse-drops/tiers` (returns `PULSE_DROP_TIERS` from config) and `GET /api/pulse-drops/nearby/{lat}/{lng}`.

### backend/app/routes/oracle.py
**Gated route:** `GET /venues/{venue_id}/oracle/premium`
**Role:** Premium Claude AI oracle with heuristic baseline. Free `GET /venues/{venue_id}/oracle` is ungated.
**Vibe+ gate:** **Present.** Calls `get_current_user(request)` → raises HTTP 401 if not logged in. Calls `_check_vibe_plus(user)` → raises HTTP 402 with header `X-Vibe-Plus-Required: true` if not Vibe+.
**Notes:** Gracefully degrades to heuristic oracle if `ANTHROPIC_API_KEY` is not set.

### frontend/src/store/vibeStore.ts (billing slice)
Billing actions are defined in `frontend/src/store/slices/authSlice.ts` and exported via `vibeStore.ts`.

**Billing actions:**
- `isVibePlus()` — Checks `user.is_vibe_plus` flag and `user.vibe_plus_expires_at` client-side; returns false if expired date is in the past
- `initializeVibePlus()` — POST `/api/subscription/initialize` with Bearer token; returns `{ authorization_url, reference }`
- `verifyVibePlus(reference)` — POST `/api/subscription/verify/{reference}` with Bearer token; on success updates local user state with `is_vibe_plus=true` and `vibe_plus_expires_at`
- `refreshSubscriptionStatus()` — GET `/api/subscription/status` with Bearer token; syncs `is_vibe_plus` and `vibe_plus_expires_at` from server

**Backend endpoints called:**
- `POST {API_URL}/api/subscription/initialize`
- `POST {API_URL}/api/subscription/verify/{reference}`
- `GET {API_URL}/api/subscription/status`

**Notes:** `isVibePlus()` is a pure client-side check against local store state — it does not make a network call. This means if the server expires a subscription, it won't be reflected until `refreshSubscriptionStatus()` is called.
