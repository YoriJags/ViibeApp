## BILL-01: Vibe+ Subscribe + Feature Gates

**Requirement:** Scout can subscribe to Vibe+ via RevenueCat (Apple IAP on iOS, Google Play Billing on Android, Paystack web) for ₦2,000/month; Oracle AI + Night Planner unlock immediately.

**Verdict:** ⚠ Partial

### Evidence

| Sub-requirement | Status | Evidence (file:line or description) |
|-----------------|--------|--------------------------------------|
| Paystack subscribe route (web) | ✓ | `backend/app/routes/subscriptions.py` — `POST /api/subscription/initialize` |
| Price = ₦2,000 (200,000 kobo) | ✓ | `subscriptions.py` line 20: `DEFAULT_PRICE_KOBO = 200000` |
| Oracle Premium gated | ✓ | `backend/app/routes/oracle.py` — `get_oracle_premium()` calls `_check_vibe_plus(user)`, raises HTTP 402 if not Vibe+ |
| Night Planner gated | ✗ | `backend/app/routes/planner.py` — `POST /api/planner/chat` has no auth check and no call to `_check_vibe_plus()` |
| RevenueCat iOS/Android SDK | ✗ | `frontend/package.json` — `react-native-purchases` not present |
| RevenueCat webhook endpoint | ✗ | `backend/app/routes/webhooks.py` — no RevenueCat event handler registered |

### Gaps (if any)
- Gap 1: Night Planner route (`POST /api/planner/chat`) is unprotected — any unauthenticated user can call the Claude AI concierge without a Vibe+ subscription. No `Request` parameter, no `require_auth`, no `_check_vibe_plus()` call.
- Gap 2: Night Planner UI (`NightPlannerModal.tsx`) has no `isVibePlus()` check — the modal can be opened and used by free users regardless of subscription status.
- Gap 3: RevenueCat iOS/Android IAP integration is entirely absent. `react-native-purchases` not installed, no API key configured, no backend webhook endpoint for RevenueCat events.

## BILL-02: 7-Day Free Trial

**Requirement:** New Vibe+ subscriber sees 7-day free trial before first charge; subscription status screen shows trial end date, feature list, and cancel link.

**Verdict:** ✗ Missing

### Evidence

| Sub-requirement | Status | Evidence |
|-----------------|--------|----------|
| trial_ends_at stored on subscription initialize | ✗ | `subscriptions.py` — `pending_subscriptions` insert does not include `trial_ends_at` field |
| Paystack start_date used to delay first charge by 7 days | ✗ | `subscriptions.py` `initialize_subscription()` — Paystack payload includes `amount`, `email`, `reference`, `currency`, `metadata`, `callback_url` only; no `start_date` parameter |
| Status endpoint returns trial_ends_at in response | ✗ | `GET /api/subscription/status` returns `is_vibe_plus`, `expires_at`, `days_remaining` only |
| Trial end date shown in UI | ✗ | `VibePlusModal.tsx` — no trial UI state, no trial end date display |
| Feature list on status UI | ✗ | No subscription status screen exists in frontend; `VibePlusModal.tsx` is a paywall, not a status screen |
| Cancel link on status UI | ✗ | No subscription management screen exists in frontend |

### Gaps (if any)
- Gap 1: No trial mechanism at all — `_apply_subscription()` unconditionally sets `expires_at = now + 30 days`. First charge is immediate.
- Gap 2: No subscription status screen (separate from the paywall modal) — no place for the user to see trial end date, feature list, or cancel link.
- Gap 3: No cancel flow — no Paystack subscription cancellation endpoint exists in the backend.
- Gap 4: The subscription model is single-charge (one-off transaction), not a recurring Paystack subscription plan — there is no `plan_code` in the initialize payload and no `subscription_code` stored, so Paystack will not auto-renew at the 30-day mark.

## Resilience Findings

### Naira Fluctuation Handling

| Check | Finding |
|-------|---------|
| Webhook verifies received amount == expected amount | No — webhooks.py routes by reference prefix only; it calls `verify_subscription_by_reference()` which re-calls Paystack's verify endpoint, and that path does NOT compare the received webhook `data.amount` against `DEFAULT_PRICE_KOBO` |
| Kobo amount stored at charge-time (snapshot) vs. re-computed from constant | Re-computed: `verify_subscription_by_reference()` calls Paystack verify to get current status, but the amount check (`data.get("amount") < price_kobo`) only exists in the user-facing `verify_subscription()` route (subscriptions.py line 311), not in the webhook path |
| Price constants: single config source vs. hardcoded in multiple places | Hardcoded in 2 places: `DEFAULT_PRICE_KOBO = 200000` in `subscriptions.py` line 20 (used as fallback); `platform_settings` DB collection used as dynamic source via `get_platform_setting("vibe_plus_price_kobo")`. Pulse Drop prices are in `config.py PULSE_DROP_TIERS` and duplicated in `PulseDropSelector.tsx` component. No merchant SaaS price constant exists. |

**Naira fluctuation gaps:**
- Gap 1: Webhook path (`verify_subscription_by_reference`) does not verify that the received `data.amount` from Paystack matches the expected subscription price. A Paystack `charge.success` event for a ₦500 transaction with a `VIIBE-PLUS-` reference prefix would activate a full Vibe+ subscription.
- Gap 2: `_apply_subscription()` reads price from `platform_settings` for revenue logging but does not validate against the charge amount — the revenue record could log the wrong price if the setting was changed after the charge.

### Network Failure Handling

| Check | Finding |
|-------|---------|
| httpx/requests timeout parameter on provider API call | Present on initialize (`timeout=15`) and user-facing verify (`timeout=10`); also present on webhook-path verify (`timeout=10`). Absent on `resolve_bank_account` and `create_transfer_recipient` in payments.py — but those are cashout helpers, not subscription flows. |
| try/except around provider API call | Present on initialize (catches `HTTPException`, `httpx.TimeoutException`, `Exception`); Present on webhook-path verify (catches general `Exception`); Absent — user-facing verify (`verify_subscription()`) catches general `Exception` but re-raises as HTTP 400 with `str(e)` — exposing raw httpx error messages to the client on network failure |
| HTTP response caller receives if provider unreachable | Initialize: HTTP 504 on timeout, HTTP 500 on general error (safe). User-facing verify: HTTP 400 with `str(e)` on any exception including network errors — this exposes internal error details (e.g., "Connection refused" or httpx exception class names) to the frontend. |

**Network failure gaps:**
- Gap 1: `verify_subscription()` (user-facing POST endpoint) catches `Exception` but raises HTTP 400 with `str(e)` as the detail. A network error contacting Paystack will expose a raw httpx exception message like `"HTTPStatusError: ..."` or `"ConnectError: ..."` to the client. Should return a sanitized HTTP 503 message like "Payment provider temporarily unavailable."
- Gap 2: No `httpx.RequestError` or `httpx.TimeoutException` specific catch in `verify_subscription()` to distinguish network failures from payment failures (both currently return HTTP 400 with str(e)).

Gaps ordered by Phase 1 success criteria impact (highest impact first).

- [ ] Night Planner frontend modal has no isVibePlus() check (affects SC-1, BILL-01)
- [ ] Merchant SaaS subscription entirely absent — no route, no DB, no Paystack Plan, no webhooks (affects SC-3, BILL-03)
- [ ] Pulse Drop purchase route has no SaaS gate (affects SC-4, BILL-04)
- [ ] Pulse Drop frontend has no SaaS upsell prompt (affects SC-4, BILL-04)
- [ ] 7-day free trial not implemented — no start_date, no trial_ends_at, subscription is immediate one-off charge (affects SC-2, BILL-02)
- [ ] No subscription status screen — user cannot see trial end date, feature list, or cancel link (affects SC-2, BILL-02)
- [ ] RevenueCat iOS/Android IAP entirely absent (affects SC-1, BILL-01)

### P2 — Important (requirement partially met; product works but incorrectly)
- [ ] Webhook path does not verify received amount matches expected price — replay attack with lower amount would activate Vibe+ (BILL-01 security)
- [ ] `verify_subscription()` returns HTTP 400 with raw exception messages on network failure — exposes internal errors to clients (BILL-01 resilience)
- [ ] Vibe+ subscription model is a one-off charge, not a Paystack recurring plan — will not auto-renew at 30 days (BILL-01)

### P3 — Minor / Deferred (noted for future phases)
- [ ] Webhook logs "Invalid Paystack webhook signature" with no IP, signature prefix, or event type — minimal diagnostic context for replay attack investigation (deferred per CONTEXT.md — minor logging improvement)
- [ ] `isVibePlus()` in frontend store is client-side only — stale subscription state until `refreshSubscriptionStatus()` is called (minor UX issue, not a security gap)
- [ ] `resolve_bank_account` and `create_transfer_recipient` in payments.py lack try/except (cashout helpers, not subscription flow — deferred)

## Recommendation for Next Steps

The billing implementation is approximately 25% complete relative to Phase 1 success criteria. BILL-01 has a solid Paystack web foundation (initialize/verify/webhook) but is missing the Night Planner gate and all RevenueCat integration. BILL-02, BILL-03, and BILL-04 are essentially unimplemented. Two resilience gaps (webhook amount validation, user-facing error exposure on verify) are minor enough to close in-phase per CONTEXT.md guidance. The Night Planner Vibe+ gate is a small targeted fix (add auth + `_check_vibe_plus` call to planner.py) also suitable for this phase. Merchant SaaS (BILL-03), free trial (BILL-02), and RevenueCat (BILL-01 iOS/Android) require new build work and are out of scope for this documentation + minor fixes phase.
